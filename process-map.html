Sourcing retry loop (N4-N6, the part IT asked about)

Plain English: Claude polls Clay for ICP contacts at the chosen account. It keeps only contacts with a verified email (phone optional, see toggle). If a pass returns fewer than 20 valid contacts, Clay is polled again with the next page / broadened titles, up to 3 passes total. If still short and the account is genuinely small, the day is topped up from the next hopper account; if zero are found, the account is parked and the run continues.

1find-and-enrich-contacts-at-company (ICP titles, US) then add-contact-data-points [Email] + get-task-context pollverified email (and ≥1 phone if require_phone on)go to pass 2
2same, next page (per_page≤100) / broadened title setsamego to pass 3
3same, next page / broadenedsameexhausted: top up from next account, or park if 0 (F2)

Failure modes

IDConditionSystem behavior
F1Tool not pre-approved (no "Run now")Unattended session stalls on first tool call. One-time operator fix.
F2Clay returns 0 after 3 passesSlack notice, gate.py set <account> todo, continue run. No crash.
F3Enrichment stuck in-progressBounded poll on get-task-context; drop unresolved contacts, proceed.
F4Apollo key lacks master scopePer-contact fields can't be written by API; CSV import (N14) is the path.
F5Any step throwsSlack one-line failure, release lock (status->todo), write run log, exit clean.
F6Sequence missing/paused at enrollApollo error surfaced; no blind retry.

Safety invariants

#Invariant
1Account locked (in_progress) before work, released on any exit.
2Unattended driver NEVER calls add_contact_ids or approve. Nothing sends without human confirm (N13) + activate (N14).
3All Apollo/Slack IDs come from a live in-session tool call; never fabricated.
4processed-replies.jsonl + active-contacts.jsonl keep the 15-min watcher idempotent.
